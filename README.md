# CS320-Software-Test-Automation

Ensuring Functionality and Security

When approaching a code with the premise of functionality and security, the things that always come to mind first and foremost are Unit testing, integration testing, and boundary testing for functionality. Then, for security, you must focus on invalid inputs, encryption, and ensuring your inner code cannot access the rest of your code without permission. Unit testing aims to test components in isolation, while integration focuses on how they interact, and boundary value testing typically goes hand in hand with security, assuring that invalid inputs are not possible.

Interpreting and Incorporating User Needs

User needs always begin with gathering their requirements, what do they need it to do? How do they want it done? The techniques often used for this are user stories, which approach the code from the user's perspective. As a user, I would like 'this' to do 'that'. Once you have this, you begin writing the code, completing it over time with many chances for feedback loops, assuring the client is allowed to give input as early and often as possible, for a cost.

How to Approach Designing Software

When designing software, I find it best to break a large-scale project into much smaller parts. You then work on each of these parts one by one and ensure it works before moving to the next. Eventually, most of the time, you will get to the part where you have to use integration; however, if you broke the smaller things apart and they had no issues when separated, you know your issue is in the integration at this point.
